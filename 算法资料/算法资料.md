# 中控杯初赛 · 算法备赛资料（选手版）

> 本文用于**固定**本届线上赛「视觉算法评测」侧的**任务类型、难度档位、标签与能力要求**，便于各队按同一口径训练模型、设计推理接口。  
> **初赛总分**通常由「视觉 API 评测」与「在线答题」两部分组成；**在线答题**以平台公示的题型为准，**不在本文展开**。  
> **与《初赛方案》的关系**：《初赛方案》中规划了**五类**视觉子题（含关键点、分割等）；**当前线上竞赛平台已实现并为判分使用的是下列三类**（`ocr` / `detect` / `classify`）。若后续赛站升级为五类，以平台「健康检查」返回的 `supported_tasks` 与赛前通知为准。

---

## 一、视觉侧总体结构（请按此规划训练）

| 维度 | 固定约定 |
|------|----------|
| 任务类型 `task_type`（本平台） | `classify`（图像分类）· `detect`（目标检测）· `ocr`（文字识别） |
| 难度档位 `difficulty` | `L1`（易）· `L2`（中）· `L3`（难），与题库一一对应 |
| 作答单元 | 每一组 `task_type` + `difficulty` 称为一个**槽位**；须完成 **3 类 × 3 档 = 9 个槽位**（每槽在规则内可多次抽图重试，取有效成绩） |
| 接口形态 | 平台经本队 **Agent 网关** 调用你方服务的 `GET /health`、`POST /infer`（JSON）；具体字段以平台《API 与评测说明》为准 |

**健康检查**：`GET /health` 的 JSON 中建议包含 `supported_tasks`，须覆盖你队承诺支持的类型，例如：`["ocr","detect","classify"]`。

---

## 二、三档难度 `L1` / `L2` / `L3`（所有题型通用）

| 档位 | 名称 | 能力侧重（训练方向） |
|------|------|----------------------|
| **L1** | 难度一 | 题面相对规整：对比度高、透视/遮挡较少；可用**传统图像处理 + 轻量模型**或小型网络达到可用精度。 |
| **L2** | 难度二 | **光照、视角、尺度**变化与一定干扰；建议**数据增强 + 经典小模型**（如轻量分类网、小目标检测头）。 |
| **L3** | 难度三 | **细粒度、密集、弱对比、类间易混**等；以**深度学习**为主，强调鲁棒性与召回，注意过拟合与误检。 |

**抽题难度**：每一槽位正式评测时，平台会从 **L1 / L2 / L3** 中按赛站配置的概率抽取一档再随机抽题图；**概率与单题满分**以平台公示的「评测配置」为准（常见参考：易 30% / 中 50% / 难 20%）。

## 三、任务类型与「题型标签」固定表

下表为选手在 **`/infer` 请求体**中会看到的 **`task_type`** 与中文含义；**请按行实现分支逻辑**，并与 `supported_tasks` 一致。

| `task_type`（固定英文，小写） | 中文题型 | 题意摘要 | 与判分强相关的输出要点 |
|------------------------------|----------|----------|------------------------|
| `classify` | 工况 / 场景分类 | 判断整图属于哪一类工况或场景 | 输出与 **`meta.class_names`** 一致的**类别标签** |
| `detect` | 工业目标检测 | 在图中定位指定类别目标 | **`meta.coord_mode` 固定为 `pixel`**：`targets`/`boxes` 的坐标均为**相对题图左上角的像素**（与题图宽高一致）；类别须在 `class_names` 内 |
| `ocr` | 表计 / 铭牌文字识别 | 读出指定区域或整图的规范文本 | 输出**字符串**；注意与 **`meta.normalize_rules`** 等一致 |

**说明**：同一 `task_type` 下，**L1 / L2 / L3** 通过**不同子题库**体现难度差异（成像复杂度、类别数、干扰等），**不改变**你在 `infer` 里应返回的**字段结构大类**（分类仍是类别、检测仍是框/点、OCR 仍是文本）。

## 四、各题型按难度的命题主轴（训练目标）

> 下列为与《初赛方案》对齐的**命题意图**摘要，便于组队分工与数据合成；**具体每张图的真值与元数据**以平台当次请求中的 `meta` 为准。

### 4.1 文字识别 `ocr`

| 档位 | 命题主轴（建议训练数据形态） |
|------|------------------------------|
| L1 | 单行、规整印刷体或数码管类读数；高对比、少粘连 |
| L2 | 多行、轻透视或阴影；需分行或整体拼接策略 |
| L3 | 强反光、污渍、小字密集；宜「检测文本行 + 识别」或端到端鲁棒模型 |

### 4.2 目标检测 `detect`

| 档位 | 命题主轴（建议训练数据形态） |
|------|------------------------------|
| L1 | 单类或少实例、朝向较正；框尺度适中 |
| L2 | 多类多实例、尺度变化与部分遮挡；含一定比例小目标 |
| L3 | 密集、弱对比、类间易混；需控制误检与漏检 |

**坐标系（固定）**：**仅像素**。`meta.coord_mode` 恒为 **`"pixel"`**；题库中评委参考框 `expected.boxes[].xyxy` 与选手返回的 **`targets[].cx` / `cy` 或 `boxes[].xyxy`** 均使用**同一幅题图**的像素坐标系（原点左上，x 向右、y 向下）。训练与推理前请先读题图实际宽高，保证输出与图像像素对齐。

**判分语义（本平台）**：赛题给出**评委参考框**；算法输出以 **「目标中心落在对应参考框内 + 类别一致」** 等规则计分。训练时可重点优化 **中心点像素精度** 与 **类别判别**。

### 4.3 图像分类 `classify`

| 档位 | 命题主轴（建议训练数据形态） |
|------|------------------------------|
| L1 | 3～5 类，类间差异大 |
| L2 | 更多类、细粒度（相似外观子类） |
| L3 | 多标签或含干扰类；需拒识或多头输出策略 |

## 五、单槽满分（与当前平台计分一致，便于定目标）

视觉模块各槽位**满分**仅与 **`task_type` + `difficulty`** 有关（自动评测得分率 0～1 再乘该满分）。**下表为当前程序内固定档位**，用于备赛估算「冲分性价比」：

| `task_type` | L1 满分 | L2 满分 | L3 满分 |
|-------------|---------|---------|---------|
| `classify` | 8 | 10 | 12 |
| `ocr` | 8 | 10 | 12 |
| `detect` | 10 | 13 | 17 |

**视觉模块总分**：九槽得分按赛站规则**折算归一**到组委会设定的视觉封顶分（如 50 分），与在线答题分数合成初赛总分；**以当年赛前公示为准**。

## 六、请求里的 `meta` 与「标签」怎么读（实现必读）

平台会在 **`/infer` 的 `meta`**（及必要时 `draw` 等字段）中给出本题可用的类别名、坐标模式、语言提示、OCR 规范化规则等。选手侧应：

1. **分类 `classify`**：使用 **`meta.class_names`**（字符串数组）作为**合法类别全集**；返回值中的标签必须落在该集合内（或与平台约定的大小写规则一致），否则可能判为无效标签得 0 分。
2. **检测 `detect`**：必须出现 **`"coord_mode": "pixel"`**（在根 `meta` 或样本 `meta` 中，与平台下发一致）。使用 **`meta.class_names`**；**`targets` 的 `cx,cy` 或 `boxes` 的 `xyxy` 均为像素**，与当幅题图宽高一致；**不得使用 0～1 归一化坐标**。
3. **文字 `ocr`**：关注 `language_hint`、`normalize_rules`（如是否去空格、英文是否忽略大小写）等；**先对预测串做同一套规范化**再与标准比对。

**注意**：`request_id`、`session_id`、`task_type`、`image` 为每题必带字段；须在约定超时内返回 JSON，且 **`ok` / `result` / `message`** 等顶层结构符合平台《API 与评测说明》。

## 七、备赛与联调检查清单

- [ ] 三套分支：`classify` / `detect` / `ocr` 均能根据 **`meta`** 解析类别与坐标规则  
- [ ] **`detect`**：确认请求里 **`coord_mode === "pixel"`**，且输出坐标与**题图像素尺寸**一致  
- [ ] `GET /health` 中 `supported_tasks` 包含上述三类  
- [ ] 为 **L1、L2、L3** 分别准备代表性样本做离线评测，避免只拟合单一难度  
- [ ] 控制单张图端到端时延，满足平台 **`infer_T_max_ms`**（每题可能不同，以请求为准）  
- [ ] 异常路径：超时、非 JSON、`ok: false` 时仍能返回可解析结构，便于平台记录原因
